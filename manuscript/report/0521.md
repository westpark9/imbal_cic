## 2. 이번 기간(0511–0521) 수행 작업

### 2.1 MATI 적용 시도 및 중단 (`code_mati.py`, `code_mati_repo.py`)

#### 배경

MATI(WSDM 2026)는 test-time에 불균형 분포를 추정하고, 각 expert의 예측을 사후 보정하는 MoE 방식이다. 핵심 아이디어는 test-time class prior를 추정한 뒤 expert 출력에 곱해 imbalance를 보정하는 것으로, routing 오류 문제를 우회할 가능성이 있어 검토 대상으로 선정했다.

#### 시도 내용

- `code_mati.py`: MATI의 prior 추정 및 expert 보정 로직을 CIC-IDS2017 환경에 맞게 이식 시도.
- `code_mati_repo.py`: MATI 공식 레포지토리 코드를 직접 재현하는 방식으로 접근.

#### 중단 이유

MATI 원 구현은 **회귀(regression) 기반**으로 설계되어 있다. 구체적으로, test-time class prior 추정 과정이 예측값의 연속 분포를 가정하는 절차를 포함한다. CIC-IDS는 이산 다중 클래스 분류(multi-class classification) 문제이므로, 이를 그대로 적용하기 위해서는 회귀를 분류로 변환하는 비자명한 설계 결정이 필요하다.

이 변환이 MATI의 핵심 아이디어(prior 보정)를 유지한 채로 가능한지 불분명하고, 변환 과정에서 MATI의 이론적 근거가 훼손될 가능성이 있어 재현 시도를 중단했다.

### 2.2 TTA(Test-Time Augmentation) 도입 시도 (`code_tta.py`)

#### 배경 및 아이디어

MATI 재현이 중단된 후, 동일하게 test-time에 작동하면서 routing 문제를 우회할 수 있는 방향으로 **SADE(NeurIPS 2022)** 방식의 TTA를 검토했다.

SADE의 핵심 수식:

```
S = (1/B) Σ_b  ŷ¹_b · ŷ²_b
```

여기서 `ŷ¹`, `ŷ²`는 동일 샘플의 두 augmented view에 대한 앙상블 예측이다. S를 최대화하면 예측의 일관성(stability)이 높아지고, Theorem 1에 따라 `I(Ŷ;Y) − H(Ŷ)`에 비례하므로 올바른 예측을 향해 수렴한다.

이를 Adam gradient ascent로 expert별 global scalar weight를 최적화한다.

#### 설계

`code_tta.py`는 이전 시도들과 달리 **배타적 분할(exclusive partition)을 사용하지 않는다.** 각 expert는 전체 데이터를 학습하되, 소속 클래스에 `focus_weight` 배의 sample weight를 부여하는 **inclusive expert** 설계를 채택했다. 모든 expert가 전체 15개 클래스에 대한 확률 벡터를 출력한다.

```
expert 훈련:
  모든 샘플로 학습 (balanced weight × focus_weight for focus classes)
  → 출력: 전체 클래스 확률 벡터 [N, 15]

TTA (SADE):
  x¹, x² = VIME perturb(x)
  S = (1/B) Σ_b ŷ¹_b · ŷ²_b  (ŷ = Σ_e w_e · p_e(x))
  Adam gradient ascent on w  →  최종 global weights
```

결과는 베이스라인 대비 개선을 확인하지 못했으나, 구조적으로 가장 일관된 설계임을 확인했다.

### 2.3 배타적 분할 + TTA 시도 (`code_tta_js.py`, `code_tta_ent.py`)

#### 배경

`code_tta.py`의 inclusive expert는 XGBoost 내부 부스팅과 focus_weight가 기능적으로 겹쳐 expert 간 차별화가 제한된다는 가설 하에, 각 expert가 담당 클래스만 학습하는 **배타적 분할(exclusive partition)** 을 시도했다.

두 가지 TTA stability 지표를 비교했다:

- `code_tta_js.py`: JS divergence 기반 per-sample per-expert stability
  ```
  stability_e(x) = 1 / (1 + (1/K) Σ_k JS(p_e(x), p_e(x'_k)))
  ```
- `code_tta_ent.py`: entropy 기반 stability
  ```
  stability_e(x) = 1 / (1 + (1/K) Σ_k H(p_e(x'_k)) / log(n_classes_e))
  ```

두 방식 모두 macro F1이 0.17–0.32로, 베이스라인(0.87)에 비해 처참하게 낮았다. tail이 아닌 majority 클래스(benign recall 57%, dos-hulk recall 21%)에서도 성능이 붕괴했다.

#### 원인 분석

배타적으로 분할된 expert는 자신의 파티션 **밖의 클래스를 훈련 데이터에서 본 적이 없다.** 따라서 파티션 외부 샘플이 입력되면, 자신이 아는 클래스 중 하나를 높은 confidence로 틀리게 예측한다. 이 예측은 원본 x와 perturbed x'에 대해 **일관되게** 틀리므로, stability 기준으로도 높은 점수를 받는다.

예시 (dos-hulk 샘플에 대한 각 expert의 행동):

| Expert | pred(x) | conf | pred(x') | conf | stability |
|---|---|---|---|---|---|
| ✓ DoS | dos-hulk | 0.9997 | dos-hulk | 0.9540 | 0.9748 |
| ✗ BruteForce | ssh-patator | 0.9972 | ssh-patator | 0.9972 | **0.9989** |
| ✗ DDoS_Scan | portscan | 0.9950 | portscan | 0.9900 | 0.9303 |

BruteForce expert가 올바른 DoS expert보다 **stability가 더 높다.** TTA는 BruteForce expert에 더 높은 weight를 부여하게 되어 오히려 틀린 expert를 선택한다.

결론: JS와 entropy 중 어떤 consistency 함수를 쓰느냐는 부차적인 문제이며, **배타적 분할과 TTA는 구조적으로 양립하지 않는다.**

### 2.4 진단 실험 (`code_tta_diag.py`)

배타적 분할 + TTA의 실패가 구조적 원인임을 세 가지 실험으로 입증했다.

#### (A) 성능 비교: Oracle routing vs JS-TTA routing

Oracle routing은 test 시 true label을 알고 있다고 가정해 owning expert를 직접 선택하는 이론적 상한선이다.

| 방법 | Macro F1 |
|---|---|
| Baseline (XGBoost) | 0.8737 |
| **Oracle routing** | **0.9507** |
| JS-TTA routing | 0.1662 |

- Oracle이 베이스라인을 상회한다 → **expert 자체의 품질은 충분하다.**
- Oracle과 JS-TTA의 gap = 0.78 → **routing, 즉 TTA가 bottleneck이다.**

#### (B) Expert behavior on x and x'

각 representative 클래스에 대해 모든 expert의 pred(x), conf(x), pred(x'), conf(x'), stability, TTA weight를 직접 집계했다. Non-owning expert들이 OOD 입력에 대해 잘못된 클래스를 confidence ~0.99로 예측하며, 이 예측이 x와 x'에 걸쳐 안정적(stability > 0.99)임을 확인했다. 그 결과 TTA weight가 owning expert와 non-owning expert 사이에서 거의 균등(≈ 1/E)하게 배분된다.

#### (C) Stability gap 분포

각 샘플에 대해 `Δ = stability_owner − max(stability_non_owners)` 를 계산했다.

- 전체 테스트 샘플의 **83.1%** 에서 Δ < 0
- 15개 클래스 중 **14개 클래스**에서 mean Δ < 0
- Global mean Δ = −0.012

즉 대부분의 샘플에서 TTA가 owning expert보다 non-owning expert를 더 안정적이라고 판단하며, 이로 인해 routing이 역전된다.

#### 결론

배타적 분할의 expert는 자신의 파티션 밖 샘플에 대해 OOD를 인지하지 못하고 높은 confidence로 일관되게 틀린 예측을 출력한다. TTA stability는 "일관되게 옳은 expert"와 "일관되게 틀린 expert"를 구분하지 못한다. 따라서 **배타적 분할을 유지하면서 TTA로 올바른 expert를 선택하는 것은 불가능하다.**

배타적 분할 위에서 TTA가 작동하려면 각 expert에 OOD 게이팅이 선행되어야 하는데, 이는 "어느 파티션에 속하는지 알아야 게이팅이 가능하고, 게이팅이 되어야 파티션을 알 수 있다"는 순환 문제로 귀결된다 — `code_mat.py`의 Mahalanobis 게이팅이 막혔던 지점과 동일한 구조다.

### 2.5 Perturbation 방식 추가 검증 (`code_tta_diag.py` 확장)

TTA stability의 실패가 VIME perturbation 방식에 특유한 현상인지 확인하기 위해, `--perturb` 옵션으로 세 가지 perturbation을 선택할 수 있도록 확장했다.

| 모드 | 설명 | 제어 인자 |
|---|---|---|
| `vime` | 각 feature를 확률 p로 다른 행의 값으로 교체 (원래 방식) | `--p_mask` |
| `gaussian` | 각 feature에 `noise_std × feature_std` 크기의 가우시안 노이즈 추가 | `--noise_std` |
| `mask` | 각 feature를 확률 p로 열 평균값으로 대체 (BERT-style masking) | `--p_mask` |

#### 결과

세 가지 perturbation 모두에서 동일한 결론이 도출된다:

| perturbation | JS-TTA Macro F1 | Oracle Macro F1 | Oracle−TTA gap |
|---|---|---|---|
| vime (p=0.3) | 0.1662 | 0.9507 | 0.7845 |
| gaussian (σ=0.1) | 0.2935 | 0.9507 | 0.6573 |
| mask (p=0.3) | 0.1751 | 0.9507 | 0.7756 |

Oracle F1은 세 경우 모두 0.9507로 동일하다 — expert 자체의 품질은 perturbation 방식과 무관하다. TTA는 어떤 perturbation을 사용해도 routing에 실패하며, macro F1이 0.17–0.29 수준에 머문다. 이는 실패의 원인이 특정 perturbation 방식이 아니라 **배타적 분할이라는 설계 자체**에 있음을 뒷받침한다.

결과 폴더: `results/20260521_163512/` (gaussian), `results/20260521_163625/` (mask)
