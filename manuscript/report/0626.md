# 0626 Experiment Log: `code_ood_tta` routing / threshold debugging

## Goal

- Diagnose why `code_ood_tta` tends to route non-tail / non-owned samples into tail experts.
- Reduce `WebBot_Rare` / `Rare_Micro` false accepts without collapsing head-class performance.
- Check whether routing structure changes can improve tail handling beyond the baseline.

## What was tried

### 1. `expert_sweep` threshold tuning

- Run:
  - `results/20260626_135752/`
- Purpose:
  - Replace `per_class_min` with threshold sweep using owned accept / benign accept / non-owned attack accept.
- Observation:
  - `WebBot_Rare` leakage was not solved.
  - `DoS_DDoS` / `Scan_Brute` became too conservative.
  - Head recalls, especially `dos-slowloris`, `dos-slowhttptest`, `ssh-patator`, dropped noticeably.
- Takeaway:
  - Global expert-level sweep did not fix the rare expert and mainly hurt already-healthy experts.

### 2. Split `WebBot_Rare` into `WebBot_Main` and `Rare_Micro`

- Early split attempts:
  - `results/20260626_141236/`, `results/20260626_142653/`
- Correct hardcoded split:
  - `results/20260626_143046/`
- Structure:
  - `WebBot_Main = bot, web-bf, web-xss`
  - `Rare_Micro = infiltration, heartbleed, web-sql`
- Observation:
  - This substantially improved some tail routing relative to the first websplit attempt.
  - However, `Rare_Micro` still behaved like an overconfident tiny expert.
  - `web-bf` / `web-xss` could still be accepted by `Rare_Micro`.
- Takeaway:
  - Splitting helped diagnose the failure mode, but did not by itself fix expert-level rejection.

### 3. Make `exp4` fallback-only

- Pre-fallback comparison point:
  - `results/20260626_144812/`
- Fallback-only structure:
  - `results/20260626_151914/`
- New routing:
  - `exp0`: benign vs attack
  - `exp1,2,3`: primary experts compete
  - if `exp1,2,3` all reject, only then evaluate `exp4`
  - if `exp4` rejects too, fallback to benign
- Observation:
  - `web-bf` / `web-xss` no longer got dominated by `Rare_Micro` as before.
  - `web-sql` got worse because `Rare_Micro` could not rescue it unless primaries all rejected.
  - `ssh-patator` remained weak because `WebBot_Main` still accepted it too often.
- Takeaway:
  - The structure reduced direct `exp4` contamination, but created a recall problem for `web-sql`.

### 4. Add fallback margin-gap condition

- Runs:
  - `results/20260626_160504/` with `fallback_margin_gap=0.5`
  - `results/20260626_160805/` with `fallback_margin_gap=1.0`
- Observation:
  - `0.5` was effectively identical to `results/20260626_151914/`.
  - `1.0` slightly reduced `fallback_accept`, but mainly converted a small portion of rare selections into `all_reject`.
  - Macro-F1 stayed essentially unchanged.
- Takeaway:
  - The bottleneck was not fallback selection itself.
  - The real problem was that `Rare_Micro` still accepted too many non-owned attack samples.

### 5. Add hard-negative threshold calibration

- Run:
  - `results/20260626_162318/`
- Mode:
  - `threshold_mode=hard_negative`
- Intended behavior:
  - Keep per-class owned accept floors.
  - Penalize validation-time benign and non-owned attack accepts during threshold choice.
- Observation:
  - Overall macro-F1 improved relative to fallback-only `per_class_min`:
    - `0.7175 -> 0.7341`
  - `ssh-patator` improved relative to fallback-only `per_class_min`, but remained much worse than pre-fallback websplit:
    - `results/20260626_151914/`: `ssh-patator f1(M)=0.3749`
    - `results/20260626_162318/`: `ssh-patator f1(M)=0.5622`
    - `results/20260626_143046/`: `ssh-patator f1(M)=0.9841`
  - `web-sql` still collapsed because `Rare_Micro` was fallback-only and primary experts did not all reject it.
- Takeaway:
  - Hard-negative thresholding helps, but does not solve the structural `web-sql` issue.
  - It also does not fully suppress `WebBot_Main` false accept on head classes such as `ssh-patator`.

## Current diagnosis

### A. Why `ssh-patator` is still weak

- Main symptom:
  - `WebBot_Main` accepts `ssh-patator` far too often after the websplit/fallback restructuring.
- Best evidence:
  - `results/20260626_162318/activation_matrix.csv`
  - `results/20260626_162318/bottleneck_by_class.csv`
- Interpretation:
  - `Scan_Brute` itself is not simply rejecting `ssh-patator`.
  - Instead, `WebBot_Main` is over-accepting it, so the sample is often routed to the wrong expert.

### B. Why `web-sql` got worse

- Main symptom:
  - `Rare_Micro` can accept `web-sql`, but is not allowed to rescue it unless all primary experts reject.
- Best evidence:
  - `results/20260626_151914/activation_matrix.csv`
  - `results/20260626_162318/activation_matrix.csv`
  - `results/20260626_162318/bottleneck_by_class.csv`
- Interpretation:
  - This is mainly a structural issue of fallback-only `Rare_Micro`.
  - `web-sql` is rare, but primary experts still accept it, so fallback is never reached.

## What did not work well

- `expert_sweep` as a global threshold fix
- fallback margin-gap alone
- making `Rare_Micro` fallback-only without any rare-override mechanism

## Recommended next step after 0626

### Priority 1

- Keep the hard-negative idea, but make it more targeted:
  - stronger negative pressure for `WebBot_Main` against confusing head classes such as `ssh-patator`
  - avoid treating all non-owned attack negatives equally

### Priority 2

- Revisit `Rare_Micro` usage for `web-sql`:
  - fallback-only is too strict
  - likely need a rare override rule, e.g. allow `Rare_Micro` to override primaries when its margin is sufficiently stronger
