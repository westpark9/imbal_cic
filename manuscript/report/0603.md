# Diagnostic Report: Why TTA-Based Routing Fails in Exclusive-Partition MoE

**Date:** 2026-06-03  
**Script:** `src/code_tta_diag.py`  
**Dataset:** CIC-IDS2017 (`data/cic2017_proc.pkl`)  
**Results:** `results/20260529_075206/`

---

## 1. 배경 및 목적

기존 MoE(Mixture-of-Experts) 접근에서 핵심 미해결 문제는 **라우팅 실패**다. 전문가(expert) 모델 자체의 품질이 충분하더라도, 테스트 시점에 올바른 전문가를 선택하는 라우터가 작동하지 않으면 앙상블 성능이 단일 XGBoost 베이스라인보다 오히려 낮아진다.

`code_tta_diag.py`는 이 실패의 메커니즘을 세 가지 증거(A/B/C)와 노이즈 민감도 스윕(D)으로 정량화한 진단 실험이다. 핵심 가설: **JS 기반 TTA(Test-Time Augmentation) 안정성 점수는 owning expert와 non-owning expert를 구별하지 못한다.**

### 2.2 전문가 파티션 (6 experts)

클래스를 공격 유형별로 배타적으로 분할한다. 희귀 클래스(IR ≥ 5,000)는 자동으로 Tail 전문가에 할당된다.

| Expert | 담당 클래스 |
|---|---|
| Benign | benign |
| DoS | dos-goldeneye, dos-hulk, dos-slowhttptest, dos-slowloris |
| DDoS_Scan | ddos, portscan |

| WebBot | bot, web-attack-brute-force, web-attack-xss |
| Tail | heartbleed, infiltration, web-attack-sql-injection |

### 2.3 TTA 방식: JS-Stability Weighting

각 전문가의 예측 안정성을 Jensen-Shannon divergence로 측정한다.

$$\text{stability}_{e,i} = \frac{1}{1 + \frac{1}{V}\sum_{v=1}^{V} \text{JSD}(p_e(x_i) \| p_e(x_i^{(v)}))}$$

여기서 $x_i^{(v)}$는 Gaussian 노이즈($\sigma = 0.1 \times \text{col\_std}$)를 추가한 augmented view, $V=5$다. 안정성 점수를 정규화하여 전문가 선택 가중치로 사용한다.

**Oracle routing (상한):** 진짜 클래스 레이블로 owning expert를 직접 선택 — 라우팅이 완벽한 경우의 성능 상한.

대표 클래스(파티션별 최다 지지 클래스)에 대해 각 전문가의 예측 및 안정성을 측정했다.

**핵심 관찰:** non-owning expert는 틀린 클래스를 예측하면서도 owning expert와 거의 동일한 confidence와 stability를 보인다.

### 3.4 (D) 노이즈 민감도 스윕

Gaussian 노이즈(σ 스케일 0.01~2.00)와 마스킹(p_mask 0.05~0.70) 두 가지 perturbation을 10회씩 반복하여 각 레벨에서의 평균 flip rate를 측정했다.

**Gaussian 노이즈 – 평균 flip rate**

| noise_std | avg flip rate |
|---|---|
| 0.01 | 0.3341 |
| 0.05 | 0.3949 |
| 0.10 | 0.4118 |
| 0.20 | 0.4249 |
| 0.50 | 0.4403 |
| 1.00 | 0.4537 |
| 2.00 | 0.4647 |

**마스킹 – 평균 flip rate**

| p_mask | avg flip rate |
|---|---|
| 0.05 | 0.0419 |
| 0.10 | 0.0787 |
| 0.20 | 0.1439 |
| 0.30 | 0.2026 |
| 0.50 | 0.3057 |
| 0.70 | 0.3841 |

**관찰:** Gaussian 노이즈는 매우 작은 σ=0.01에서도 flip rate가 0.33을 초과한다. 이는 모든 전문가가 perturbation에 함께 불안정해지며, 안정성 차이(Δ)가 노이즈 수준에 관계없이 일정하게 낮다는 것을 의미한다. 마스킹은 낮은 p_mask에서 flip rate가 작지만, 이 경우 모든 전문가가 동시에 안정적이라 Δ가 여전히 0에 수렴한다.

## 4. 실패 메커니즘 해석

### 4.1 구조적 원인: Partition-Agnostic Feature Space

CIC-IDS2017의 70개 특성은 공격 유형을 구분하도록 설계된 것이 아니다. Benign 트래픽과 공격 트래픽을 구분하는 특성들이 대부분이며, 공격 유형 간 특성 분포가 크게 겹친다. 따라서:

- 배타적 파티션 전문가들은 자신의 파티션 데이터로만 학습했음에도, **다른 파티션 데이터에도 높은 confidence로 예측**한다 (틀리지만 자신감 있게).
- 전문가가 OOD(Out-of-Distribution) 입력을 받을 때 confidence를 낮추지 않는다.

### 4.2 TTA의 전제 조건 미충족

JS-stability 기반 TTA는 다음 전제를 가정한다:

> "owning expert는 자신의 클래스에 안정적이고, non-owning expert는 불안정하다."

그러나 실제로:
- Non-owning expert는 OOD 입력에서 **틀린 클래스를 일관되게** 예측한다 (높은 stability).
- Owning expert는 augmented view에서 예측이 흔들리는 경우가 있다 (낮은 stability).
- 결과적으로 stability ≈ 1/E (균등 가중치)로 수렴하여 라우팅이 사실상 랜덤이 된다.

### 4.3 XGBoost의 특성

XGBoost는 트리 기반 모델로, 특성 공간을 hard region으로 분할한다. 한 파티션으로 학습한 모델이 다른 파티션 샘플에 대해 낮은 confidence를 보이려면, 해당 파티션 샘플이 학습 시 본 leaf node에 매핑되지 않아야 한다. 그러나 CIC-IDS2017의 특성 분포가 겹치기 때문에 이 조건이 충족되지 않는다.

## 5. 결론

`code_tta_diag.py`의 세 가지 증거는 같은 결론을 가리킨다:

> **TTA 기반 stability 점수는 exclusive-partition MoE에서 owning expert를 신뢰성 있게 식별하지 못한다. 이는 전문가 품질의 문제가 아니라, non-owning expert가 OOD 입력에서 높은 confidence를 유지하는 XGBoost의 특성에서 기인한다.**

- Oracle routing의 Macro F1(0.9507)이 베이스라인(0.8737)을 크게 상회하므로, **전문가 모델 자체는 충분히 유효**하다.
- 라우팅만 해결되면 MoE가 베이스라인을 이길 수 있다.
- 따라서 다음 연구 방향은 stability 기반 라우팅 대신, **학습 기반 라우터** (예: 클래스 소속 확률을 직접 예측하는 메타-분류기)나 **Gating Network**를 사용하는 방향이 합리적이다.
